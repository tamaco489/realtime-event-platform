# PR のルール

## タイトルのフォーマット

```text
[ラベル] [issue-番号] [変更種別] 関連 Issue のタイトル
```

例:

```text
[backend] [issue-10] [feat] EventBridge へのイベント Publish 処理を追加
```

| 要素       | 説明                                                                          |
| ---------- | ----------------------------------------------------------------------------- |
| ラベル     | `labels.md` を参照して選択する                                                |
| issue 番号 | 関連 Issue 番号 (`issue-XX` 形式)。Issue がない場合は省略                     |
| 変更種別   | commit-types.md の type と同じ (`feat`, `fix`, `docs` など)                   |
| タイトル   | 関連 Issue のタイトルをそのまま使用。Issue がない場合はコミット群から自動生成 |

## 本文のフォーマット

- `.github/PULL_REQUEST_TEMPLATE.md` のテンプレートに従う
- 文中の句点 (。) の後に改行を入れる
- 重要な箇所・特に注意してほしい場所には `> [!IMPORTANT]` または `> [!NOTE]` で強調表示する

## 本文の基本方針

- **git diff を見ればわかることは書かない** — 変更ファイル・行数・コードの内容は省略する
- **なぜ・何を・どう変えたかを端的に書く** — 実装の背景と意図を伝える
- **5W1H を意識する** — 読んだ人が変更の全体像を把握できるようにする

## 5W1H の適用指針

| 要素  | 問い                   | 記述例                                              |
| ----- | ---------------------- | --------------------------------------------------- |
| Why   | なぜこの変更が必要か   | 既存の設計では〜という問題があったため              |
| What  | 何を変えたか           | 〜を〜に変更した / 〜を追加した                     |
| Who   | 誰が影響を受けるか     | フロントエンド側の実装に影響する                    |
| When  | いつ適用・有効になるか | 次回デプロイ以降 / この PR のマージ後               |
| Where | どこに変更があるか     | backend-api の EventBridge 処理 / infra の CDK 設定 |
| How   | どのように解決したか   | 〜という方針で〜を実装した                          |

全要素を必ず書く必要はない。変更の性質に応じて必要な要素だけ選ぶ。

## 書かないこと

- コードの内容そのもの (`if err != nil { return err }` のような記述)
- 変更ファイルの列挙 (git diff・コミット一覧で確認できる)
- 自明な変更理由 (「バグがあったので修正した」など)
- 実装の詳細手順 (コードを読めばわかる)

## 書くこと

- 変更の背景・動機 (なぜこの PR が必要か)
- 設計上の判断とその理由 (なぜこのアプローチを選んだか)
- レビュアーが注意すべき点・影響範囲
- 既知の制約・今後の課題があれば簡潔に
